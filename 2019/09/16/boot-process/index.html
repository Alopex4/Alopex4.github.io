BIOS

  • 什么是BIOS
    • BIOS是嵌入在计算机主板芯片上的一个固件
      • 固件: 写入于硬件的芯片中, 为硬件服务
  • 为什么需要BIOS
    • 计算机启动后, 硬件是否可用需要检查机制 (BIOS提供POST功能)
    • 选择通过哪个存储介质进行启动需要管理机制 (BIOS提供启动优先级启动顺序)
    • 完成自身工作后, 控制权移交 (BIOS提供控制权转移)
    • 程序运行时, 屏蔽不同I/O设备的差异 (BIOS提供I/O设备抽象层)
  • BIOS工作流程
    1. 检查执行CMOS中的设定和自定义设置
      • CMOS: 是一种RAM存储,需要电子为其供电, 掉电后数据清空
      • 自定义设置: BIOS密码等
    2. 加载中断处理程序和设备驱动程序
      • 中断处理程序: 硬件组件和操作系统之间的转换器: 如当按下键盘后, 信号将发送给键盘中断处理程序
      • 设备驱动程序: 识别基本的硬件信息/操作控制特定的硬件: 键盘, 鼠标, 硬盘等
    3. 初始化寄存器和电源管理
    4. 提供电源启动自我检查(POST)
    5. 显示提示信息
      • 提示信息: 开机提示: 如’Press __ to Enter SETUP’
    6. 选拔启动设备(bootstrap)
      • 启动设备: 用于加载操作系统的设备
      • 常见的加载设备: CD, 硬盘, U盘, 网络等
    7. 移交控制权
      • 控制权: 控制权被转交回CPU

SMPS (switched-mode power supply)

  • 按下启动/重启按钮后, 计算机的供电将通过SMPS进行供给
    • 什么是SMPS
      • SMPS: 将交流电转换为稳定直流电的电子装置
    • 为什么需要SMPS
      • 家用插座一般提供220V交流电, 而计算机需要使用直流电驱动
      • 因此交流电必须转换为直流电, 普遍的转换方式有线性调节和开关调节两种
      • 相对而言, 开关调节对功率损耗较低, 因此被大量应用于计算机电源中
    • 形象理解稳定电源
      • 场景: 在一根流量直径30CM的大水管中, 提取到流量直径为5CM的小水管中
      • 解决:
        • 线性电源: 使用一个转接阀门(晶体管), 并将阀门开启到较小位置 (阀门因此需要承受很大的压力 表现为热能散失)
        • 开关调节: 将大水管的水接入到一个大水桶中, 小水管接入到大水桶, 开关大水桶与打水管连接阀门, 保证水桶内有水和防止桶内水溢出
  • CPU将被唤醒, 执行首条指令, 将控制权移交给BIOS

POST (Power On Self Test)

  • BIOS检查并执行CMOS上的设定和自定义设定…
  • BIOS进行POST检查
    • 检查所需的硬件核心硬件, 如CPU, 内存, 显卡等
      • 由于在POST检查在显卡初始化之前, 因此当发生错误时BIOS将通过蜂鸣器发出du..du..声进行报警
      • 短促的du一声, 一般意味着正常启动
    • POST检查方式
      • BIOS会记录上一次计算机启动时的所有设备
      • 在当前的启动中, BIOS将会依据上一次的记录, 发送电脉冲到每个设备(记录)
      • 如设备回应电脉冲, 该设备将被认为工作良好
      • 如设备不回应电脉冲, 该设备就被认为无法工作或被移除

bootstrap

  • BIOS通过CMOS中用户指定的硬件加载顺序
  • 读取硬件设备相应的启动或引导工具
    • 如硬件加载序列中, 无法找到操作系统, 将显示提示信息’No boot device found’
  • 将控制权交还CPU

MBR

  • CPU取得控制权后, 将对启动设备的MBR进行加载
    • 启动设备: 此处定义为硬盘

MBR简介

  • 什么是MBR
    • MBR是硬盘中记录了操作系统引导方式和分区表等信息的第一个区块(512Bytes)
  • 为什么需要MBR
    • 对于BIOS
      • bootstrap中需要寻找操作系统进行引导 (MBR提供magic number标识硬盘是否有操作系统)
    • 对于操作系统
      • 如何进行引导 (MBR提供boot code协助)
      • 引导硬盘以及其他硬盘信息 (MBR提供分区表)
  • MBR是如何建立的
    • 当硬盘分区完成时, MBR将被创建
  • MBR存放在哪里
    • MBR位于硬盘的第一个扇区上, 不属于硬盘中任何分区
    • MBR具体物理地址为: 0 Cylinder, 0 Header, 1 Sector

MBR信息查看

  • 查看扇区的前512Bytes

  • 0000 0000 - 0000 01bd –> 引导代码 (446Bytes)

  • 0000 01be - 0000 01fe –> 分区表 (64Bytes)

  • 0000 01fe - 0000 01ff –> 结束标记 (2Bytes)

MBR简要解析

  • 引导代码

    • 内容: jump to boot program, disk parameters, boot program code, disk signature
    • 目的: 找到引导分区进习启动
    • 过程:
      • 找到活动分区 (0x0800 标识的bootable分区)
      • 查询是否还存在其他标记为bootable的分区

  • 分区表

    • 内容: status, starting head, staring sector, staring cylinder, partition type, ending head, ending sector, ending cylinder, relative start sector, total sectors
      • status: 标识磁盘是否为’bootable’
      • partition type: 分区类型
        • 0x00: 空白
        • 0x82: swap
        • 0x83: linux
        • 0x8e: LVM
    • 目的: 提供系统分区状态信息
    • 补充:
      • 由于MBR使用4Byte存储扇区总数, 最大表示为2^32个扇区
      • 磁盘扇区固定大小为512k(2^9)
      • MBR最大磁盘容量大小 512K * 2^32 –> 2^41 –> 2TB

  • 结束标记(魔术码)

    • 内容: 0x55aa
    • 目的: 表示MBR的结束

GRUB

  • 通过MBR可以找到活动分区以及引导程序
    • 活动分区: 标记为bootable的分区, 在分区表状态中表现为0x80
    • 引导程序: 引导系统启动的软件, 如GRUB(本文例子), lilo
  • GRUB选择菜单出现

GRUB简介

  • 什么是GRUB
    • GRUB是一种引导程序, 负责加载系统内核, 引导系统启动
    • 引导程序软件: GRUB, lilo等
  • 引导程序常用场景
    • linux中用户安装多个内核时, 引导程序提供界面让用户选择
    • 计算机安装用多个系统时, 引导程序提供界面让用户进行选择
  • GRUB的优势
    • 支持LBA(Logical Block Addressing Mode)
    • 灵活加载操作系统 (提供加载选项参数)
    • 引导只需提供硬盘分区号和内核文件名而非硬盘地址
    • 可支持多类型操作系统

GRUB三个阶段

  • stage1
    • MBR上存在GRUB第一阶段的代
    • 由于引导需要做的工作不少, 难以仅用446Byte存储
    • stage1的任务是将控制权交给下一stage(1.5或2)
  • stage1.5
    • stage1.5是被stage1调用的
    • 仅当系统硬件要求时, 才会调用stage1.5
    • 该阶段文件名字格式: e2fs_stage1_5, fat_stage1_5
    • stage1.5将会加载stage2
  • stage2
    • 运行GRUB的主体代码
    • 加载配置文件
    • 显示GRUB 菜单
    • 启动用户选择的系统

Kernel

  • 用户选择系统
  • GRBU转移控制权给内核
  • 探测可识别的所有硬件设备
  • 挂载initramfs, 载入适当(必须)的驱动
    • 适当: 开机过程中所最需要的核心模块 如USB, RAID, LVM, SCSI 等驱动程序
    • 释放initramfs, 挂载实际root目录
  • 以只读方式加载根文件系统

Initramfs简介

  • 基本信息:
    • 全称: Initial RAM Disk 或 Initial RAM Filesystem
    • 文件名: /boot/initrd 或 /boot/initramfs
    • 特性: 可被bootloader加载到内存中
  • 功能: 提供一个可执行的程序， 通过该程序载入开机过程中所必要的核心模块
    • 必要: initramfs是被加载到内存中的, 其文件大小受限. 非必要的驱动如蓝牙, 可在根目录挂载后添加
    • 核心模块: USB, RAID, LVM, SCSI等文件系统与磁盘接口的驱动程序

Systemd

  • 内和运行用户空间第一个应用程序/sbin/init (指向/sbin/lib/systemd/systemd)

init引入

  • 背景:
    • 仅仅将内核运行起来是毫无实际用途的, 必须由init将系统代入可操作状态
    • 如启动shell后, 便有了人机交互, 计算机可执行一些预订程序完成有实际意义任务
    • 或启动X图形系统以便提供更佳的人机界面, 更加高效的完成任务
  • 关系:
    • systemd,sysvinit和UpStart相类似,是系统的第一个进程
    • 同属于init系统, 负责产生其他所有的用户进程
    • systemd是目前较为流行且先进的init工具

Systemd信息

简介与特点

  • 设计目标是克服sysvinit固有的缺点, 提高系统的启动速度
  • Systemd 的很多概念来源于苹果Mac OS操作系统上
  • 重要特性
    • 同SysVinit和LSB init scripts兼容
      • 系统中已经存在的服务和进程无需修改
      • 降低系统向systemd迁移成本, 使systemd替换现有init成为可能
    • 更快的启动速度
      • 尽可能启动更少的进程
      • 尽可能将更多进程并行启动
    • systemd提供按需启动能力
      • 只有在某个服务被真正请求的时候才启动它
      • 当该服务结束systemd可以关闭它, 等待下次需要时再次启动它
    • 采用Cgroup特性跟踪和管理进程的生命周期
      • 当停止服务时, 通过查询CGroup, systemd可以确保找到所有的相关进程，干净地停止服务
      • CGroup已经出现了很久，它主要用来实现系统资源配额管理
    • 启动挂载点和自动挂载的管理
      • Systemd兼容/etc/fstab文件, 可以继续使用该文件管理挂载点
      • Systemd 内建了自动挂载服务，无需另外安装autofs服务以实现动态挂载点
    • 实现事务性依赖关系管理
      • 挂载NFS和启动网络的工作存在天生的先后依赖关系，无法并发执行
      • 对于这些任务, systemd维护一个”事务一致性”的概念，
      • 保证所有相关的服务都可以正常启动而不会出现互相依赖, 不至于死锁的情况
    • 对系统进行快照和恢复
      • Systemd快照提供了一种将当前系统运行状态保存并恢复的能力
    • 日志服务
      • systemd自带日志服务journald
      • 用二进制格式保存所有日志信息, 用户使用journalctl命令来查看日志信息
      • 日志是除错和监控系统的核心功能，因此它自己不能再产生问题
      • 添加和浏览 日志 非常快
      • 日志数据文件需要较小
      • 让无法检测的修改不再可能

基本概念

  • 单元的概念

    • 系统初始化个过程中的每一步都被systemd抽象为一个配置单元(unit)
    • 一个服务是一个配置单元
    • 一个挂载点是一个配置单元
    • 一个交换分区的配置是一个配置单元
      名称 代表 典型
      service 后台服务进程 mysqld
      socket 系统和互联网中的一个套接字 nscd.socket
      device 存在于Linux中的设备 -
      mount/automount 文件系统挂载相关的服务 网络的自动挂载
      path 侦测特定文件或目录类型 打印服务
      timer 定时器配置单元 取代了atd、crond等传统的定时服务
      target 执行环境类型 一群unit的集合

  • 依赖关系

    • 存在有些任务, 它们之间存在天生的依赖
    • 配置单元定义文件中的关键字来描述配置单元之间的依赖关系
      • 如unit-A依赖unit-B
      • 在unit-B的定义中用"require A"来表示

  • 事务

    • 存在循环依赖, 那么systemd将无法启动任意一个服务
    • systemd将会尝试解决这个问题
      • required是强依赖
      • want是弱依赖
      • systemd将去掉wants指定的依赖尝试打破循环

  • Target 和运行级别

    等级 systemd target note
    0 runlevel0.target, poweroff.target 关闭系统
    1, s, single runlevel1.target, rescue.target 单用户模式
    2, 4 runlevel2.target, runlevel4.target, multi-user.target 用户定义/域特定运行级别。默认等同于 3。
    3 runlevel3.target, multi-user.target 多用户，非图形化。用户可以通过多个控制台或网络登录。
    5 runlevel5.target, graphical.target 多用户，图形化。通常为所有运行级别 3 的服务外加图形化登录。
    6 runlevel6.target, reboot.target 重启
    emergency emergency.target 紧急 Shell

小结

  1. local-fs.target, swap.target: 挂载本机/etc/fstab里面所规范的文件系统和swap
  2. sysinit.target:侦测硬, 载入所需要的核心模块等动作

补充延伸

固件(firmware) vs 驱动(device drvier)

固件

  • 固件是软件的一种
  • 固件在设计上面向硬件
  • 固件是硬件的一部分, 运行在硬件板载的嵌入式芯片上
  • 开发背景是, 硬件厂家为提高硬件的使用便利性
  • 开发出固件出来, 让硬件和软件的交换变得简化
  • 示例:
    • 洗衣机内置电脑程序
    • 计算器里面的操作系统

驱动

  • 驱动是软件的一种
  • 驱动的设计面向操作系统
  • 驱动从属于操作系统, 运行在CPU上
  • 开发背景是, 世界上存在这各种个样的操作系统(内核), 各自有不同硬件操作方式
  • 为操作系统开发特定的启动, 让操作系统识别硬件发挥功能
  • 示例
    • oracle database, .net driver

关联

  • 固件较为固定和操作系统无关, 较少升级
  • 驱动和操作系统相关, 升级频率较高
  • 固件是基础, 驱动是不同操作系统对硬件功能的延伸
  • 仅当某硬件只适配单一操作系统时, 且该操作系统永不修改, 固件和驱动就基本融为一体

GRUB 延伸知识

GRUB文件信息介绍

  • 查看配置相关文件

  • GRUB的核心功能之一是加载硬盘中的内核, 使得内核能驱动整个系统的硬件

  • 因此GRUB必须识别硬盘

  • GRUB识别硬盘的语法

    • (hd0,1) # 一般的默认语法，由 grub2 自动判断分区格式
    • (hd0,msdos1) # 此磁盘的分区为传统的 MBR 模式
    • (hd0,gpt1) # 此磁盘的分区为 GPT 模式
      • 说明1: 硬盘代号以小括号 ( ) 包起来
      • 说明2: 每颗硬盘的第一个 partition 代号为 1 ，依序类推

grub.cfg文件解析

  • grub.cfg 文件位置 /boot/grub2/grub.cfg
  • GRUB并不推介直接修改该文件
  • GRUB推介修改特定配置文件, 使用grub2-mkconfig生产grub.cfg文件
    • 特定文件: /etc/default/grub, /etc/grub.d
  • 查看文件内容

grub配置文件

环境配置文件 /etc/default/grub

  • 倒数时间参数: GRUB_TIMEOUT
    • 这个设置值相当简单，后面就是接你要倒数的秒数即可, 如要等待30秒
    • 这边改成“GRUB_TIMEOUT=30即可！如果不想等待则输入0
    • 如果一定要使用者选择，则填-1即可
  • 是否隐藏菜单项目: GRUB_TIMEOUT_STYLE [menu, countdown, hidden]
    • menu: 默认菜单
    • countdown: 仅显示剩余的等待秒数(无菜单)
    • hidden: 隐藏所有
  • 讯息输出的终端机模式: GRUB_TERMINAL_OUTPUT
    • 指定输出的画面应该使用哪一个终端机来显示
  • 默认开机菜单项目： GRUB_DEFAULT [save, id, title..]
    • 指定要用哪一个菜单 (menuentry)来作为默认开机

菜单创建的脚本 /etc/grub.d

  • grub2-mkconfig将调用/etc/grub.d/文件创建grub.cfg

sysvinit vs UpStart

sysvinit

  • 历史现状
    • sysvinit是system V风格的init系统, 源于System V系列UNIX
    • 大多数Linux发行版的启动服务管理是和System V相兼容的, 被称为sysvinit
    • Linux 主要应用于服务器和PC机的时代，SysVinit运行非常良好，概念简单清晰
    • 配置主要依赖于Shell脚本
    • 致命缺点是启动太慢
  • 概况
    • 运行级别: 通常在/etc/inittab文件中定义了各种运行模式的工作范围
    • 运行顺序:
      • sysvinit需要读取/etc/inittab文件
      • 获得以下信息
        • 系统需要进入的runlevel
        • 捕获组合键的定义
        • 定义电源fail/restore脚本
        • 启动getty和虚拟控制台
      • 顺序执行文件
        • /etc/rc.d/rc.sysinit
        • /etc/rc.d/rc和/etc/rc.d/rcX.d/(X 代表运行级别 0-6)
        • /etc/rc.d/rc.local
        • X Display Manager(如需要)
  • 小结
    • 优点:
      • 概念简单
        • 开发人员只需编写启动和停止脚本
        • 将service添加/删除到某个runlevel时, 只需要执行创建/删除软连接文件即可
      • 确定的执行顺序
        • 脚本严格按照启动数字的大小顺序执行, 一个执行完毕再执行下一个
    • 缺点:
      • 启动速度过慢
        • 串行执行脚本, 无法适应新型复杂的IT环境

Upstart

  • 简介
    • UpStart源自于Ubuntu发行版
    • 在linux越发流行后, 将linux用作桌面版本的ubuntu发现sysvint不适合笔记本环境
    • UpStart基于事件机制, 如U盘插入USB接口后,udev得到内核通知发现设备, 这就是一个新的事件
    • 由于事件机制, 加快了系统启动时间
  • 特点
    • 采用事件驱动模型
      • 更快地启动系统
      • 当新硬件被发现时动态启动服务
      • 硬件被拔除时动态停止服务
    • UpStart可以很好地应用在桌面或者便携式系统中, 处理这些系统中的动态硬件插拔特性
  • Upstart概念和术语
    • job
      • 一个工作单元, 用来完成一件工作
      • 如启动一个后台服务, 或者运行一个配置命令
      • 可以理解为sysvinit中的一个服务脚本
      • 分类
        • task job –> 一定时间内会执行完毕的任务, 比如删除一个文件
        • service job –> 后台服务进程, 比如apache httpd
        • abstract job –> UpStart内部使用
    • event
      • 是一个事件
      • 以通知消息的形式具体存在
      • 一旦某个事件发生了Upstart就向整个系统发送一个消息
      • 分类
        • Signals –> 事件是非阻塞的，异步的
        • Methods –> 事件是阻塞的，同步的
        • Hooks –> 事件是阻塞的，同步的 (介于Signals与Methods之间)
    • job和event协同
      • 系统初始化时init进程开始运行, init进程自身会发出不同的事件
      • 这些最初的事件会触发一些工作运行
      • 每个工作运行过程中会释放不同的事件，这些事件又将触发新的工作运行
